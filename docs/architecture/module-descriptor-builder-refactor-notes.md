# Module Descriptor Builder Refactor Notes

Temporary working notes for the descriptor/capability refactor. Keep this file until the refactor stabilizes, then either promote it into ADR/docs or fold the useful pieces into existing architecture docs.

## Direction

- Keep `ModuleDescriptor` as the stable metadata root: name, schema, admin surface, and a list of capability features.
- Prefer polymorphic `ModuleDescriptorFeature` records over adding new root properties for every optional capability.
- Author descriptors through a builder so module metadata reads like intentional declarations instead of positional constructor bags.
- Keep root module metadata in `Shared.Modules`, backed only by `Shared.Naming`.
- Keep permission metadata in `Shared.Authorization`; it is an optional authorization/admin capability, not part of the neutral module root.
- Put capability-specific builder/read extensions beside the capability:
  - authorization owns permission descriptor extensions;
  - messaging owns published-event and subscription descriptor extensions;
  - caching owns cache metadata descriptor extensions;
  - tasks owns task metadata descriptor extensions.
- Keep runtime composition explicit. Metadata still does not auto-register modules, handlers, consumers, workers, endpoints, caches, or admin surfaces.

## External Sanity Check

- Microsoft's clean architecture guidance still frames infrastructure as depending on application/core abstractions rather than the other way around: https://learn.microsoft.com/en-us/dotnet/architecture/modern-web-apps-azure/common-web-application-architectures
- Current modular-monolith writeups still converge on strict module isolation with communication through contracts/events instead of shared internals or cross-module persistence. The implemented split follows that direction by moving messaging, caching, task, observability, naming, and module metadata contracts out of the generic application project.

## Current Weak Spots To Fix

- Keep future generic naming or identifier-syntax helpers in `Shared.Naming` unless they are clearly owned by one capability. Shared feature packages should not depend on messaging, tasks, caching, API, or domain only to reuse validation.
- `Shared.Application.Composition` is intentionally limited to constrained application registration. Keep new feature-specific contracts out of it unless they are truly application-layer composition basics.
- `Shared.Application.Events` is intentionally narrow: domain-event handler/dispatcher contracts only. Do not add integration-event, outbox, or runtime dispatch concerns there.
- `Shared.Pagination` is intentionally dependency-free. Keep it to request-bound paging value types; query contracts still belong to module application projects.
- Tests may still compose the `Shared.Infrastructure` facade when verifying host-level defaults, but optional adapter tests should prefer their narrow infrastructure registration entry points.

## Deliberate Compromises

- Keep `Shared.Infrastructure` as a host-level facade over the baseline runtime adapters. The concrete implementations live in their narrower infrastructure packages so optional adapters can reference only the slices they need.
- Keep public names explicit and string-backed; no assembly scanning or magic registration in this slice.
- Use builder/extensions rather than attributes for now. Attributes may be useful later for low-risk docs generation, but they would need tests proving no runtime composition depends on reflection.
- Keep optional capability-to-CQRS bridges out of the capability core. `Shared.Tasks.Cqrs` is the pattern to copy if caching, messaging, or future capabilities need helpers that depend on the application dispatcher.
- Allow runtime adapter registration methods to compose narrower shared infrastructure they require, such as runtime clock/id registration or CQRS pipeline registration, while keeping module application/domain projects guarded against those adapter packages.
- Keep `Unknown = 0` as the default for public contract/domain/shared semantic enums such as `CacheScope` and `ModuleTaskKind`, and reject unknown or undefined values at module boundaries. If a future capability needs richer compatibility semantics, use one documented smart-enum/code-list pattern with tests instead of per-module enum magic.

## Follow-Up Audit Targets

- Keep `eng/new-module.ps1` covered by architecture/developer-experience tests whenever descriptor authoring conventions change.
- Watch whether the facade remains useful as a host convenience. If no host-level composition value remains, a future cleanup could remove `Shared.Infrastructure` entirely and have hosts call the narrow runtime adapters explicitly.
- Watch `Shared.Naming` scope. It now intentionally owns low-level identifier syntax, including tenant ids, to avoid dragging domain or tenancy runtime packages into unrelated shared features. Generic non-naming helpers should get a narrow package like `Shared.Numerics` instead of expanding naming into a dumping ground.
- Remaining `Shared.Infrastructure` use is intentionally limited to real hosts, the admin CLI integration host harness, and tests for the facade itself. If another optional adapter test starts using the facade, prefer proving the narrower adapter composition instead.
- Revisit task handler registration only if a real use case needs the same `(module, task, payload version)` on multiple worker groups. The current model treats worker group as routing metadata that must match the descriptor, not as part of handler identity.
